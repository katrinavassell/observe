Key Flows

Cost + Revenue Calculation

Cost Calculation Automatic
1
Event includes model + inputTokens + outputTokens
2
Observe looks up per-token pricing from OpenRouter pricing tables (updated on boot)
3
cost = inputTokens * inputPrice + outputTokens * outputPrice
4
If costAmount is sent explicitly, it overrides auto-calculation
OK
Cost shown per event, per customer, per feature, per model
Revenue Attribution 3 Sources
1
Explicit: revenueAmount sent in the event payload. Highest priority.
2
Feature Pricing: Rules in feature_pricing table (e.g. blog_post = $0.05/event). Applied when revenueAmount is omitted.
3
Stripe Subscription: MRR from customer's active subscription. Allocated differently by type:
-
Flat subscription → MRR / 30, allocated per day (revenue_source: "allocated")
-
Metered/usage-based → exact per-unit revenue (revenue_source: "per_unit")
-
Tiered/graduated → tier lookup by month-to-date usage (revenue_source: "tiered")
OK
Revenue shown per customer in Customers page. Stored in observe_events.revenue_amount.
Margin Calculation Derived
1
margin = revenue - cost per customer (summed across all events)
2
margin_pct = (revenue - cost) / revenue * 100
3
If revenue = 0 and cost > 0 → margin = -100% (unprofitable)
4
If revenue = 0 and cost = 0 → margin = 0% (no data)
OK
Drives health scores and cohort assignment (champion, at_risk, unprofitable, etc.)
Health Score 0-100
1
Weighted: margin_pct (60%) + active_days_30d (40%)
2
Cohort assignment: margin >= 50% + 10+ active days → Champion
3
margin < 0 → Unprofitable | inactive 30+ days → Inactive | cost rising > 50% WoW → Rising Cost
OK
Shown on Customers page with color-coded badges

Core Flows

Signup POST /auth/signup-complete
1
User signs up via Supabase Auth (client-side)
2
Frontend calls POST /auth/signup-complete
3
INSERT into users (email, visitor_id from JWT)
4
INSERT into accounts (email, password_hash='supabase-managed', visitor_id)
5
INSERT into user_accounts (role='owner', status='active')
6
INSERT into sdk_api_keys (with account_id, encrypted_key)
OK
User sees API key on Data Sources page. Key has non-null account_id.
Event Ingest POST /api/events/ingest
1
Client sends Bearer token + events array
2
Lookup sdk_api_keys by SHA-256 hash of token
3
Read account_id directly from sdk_api_keys row
4
If null: fallback to user_accounts lookup. If still null: return 500
5
INSERT into observe_events with account_id. Cost auto-computed from model + tokens.
6
Auto-create feature_definitions for new feature keys
7
Fire checkAlerts(accountId) — no re-resolution needed
OK
Event visible in Events page. Idempotency via UNIQUE(account_id, idempotency_key).
Stripe Sync POST /integrations/stripe/sync
1
User enters Stripe API key in Data Sources
2
Key encrypted + stored in integrations (account_id, provider='stripe')
3
Sync pulls customers, subscriptions, products, prices from Stripe API
4
DELETE existing data for account, INSERT fresh with account_id
5
Revenue events created in observe_events (source='stripe')
6
If 0 customers found: warning returned (wrong key?)
OK
Customers show MRR. Margin = revenue - cost visible on dashboard.
Team Invite POST /team/join/:token
1
Owner copies invite link from Team Settings
2
Invitee clicks link, signs up or logs in
3
Find org by invite_token, resolve inviter's account_id
4
INSERT into organization_members + user_accounts (role='admin')
5
Invitee's own account preserved (ON CONFLICT DO NOTHING on visitor_org_map)
OK
Both accounts visible in account switcher. Invitee sees inviter's data.
Alerts checkAlerts(accountId)
1
Events ingested → checkAlerts fired with resolved accountId
2
Query alert_rules WHERE account_id AND enabled
3
Evaluate metric query (daily_cost, margin_percent, etc.)
4
Compare value against threshold with operator (gt/lt)
5
If triggered: send email (Resend) and/or webhook (Slack-compatible)
6
Write to alert_history. Update last_triggered_at for cooldown.
OK
Alert fires once per cooldown period. History visible on Alerts page.
Invite: New User (no account) Scenario A
1
Invitee clicks invite link, lands on /join/:token
2
Not logged in → redirected to signup
3
Signup creates: users + accounts + user_accounts (owner of their own account)
4
After signup, join handler fires: adds invitee to organization_members
5
Adds invitee to user_accounts for inviter's account (role=admin)
OK
Invitee has 2 accounts: their own (owner) + inviter's (admin). Both in account switcher.
Invite: Existing User (has account) Scenario B
1
Invitee clicks invite link, already logged in
2
Join handler finds org by token, resolves inviter's account_id
3
INSERT into organization_members (if already member, reactivates)
4
INSERT into user_accounts with ON CONFLICT DO UPDATE SET status='active'
5
visitor_org_map uses DO NOTHING — invitee's own org mapping preserved
OK
Invitee's own account untouched. Inviter's account added to switcher.
