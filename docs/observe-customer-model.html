Customer Model: Two Layers

Observe has two distinct "customer" concepts that must not be conflated.

Layer 1 — Observe's Customers

Companies that use Observe

The people who sign up, log in, and pay Tanso for Observe.

TableKey Columns
usersemail, visitor_id, stripe_customer_id, stripe_plan
accountsname, clerk_org_id, stripe_customer_id, stripe_plan
user_accountsuser_id, account_id, role

Stripe context: Tanso's own Stripe account.
users.stripe_customer_id = this user as a paying customer of Tanso.
accounts.stripe_plan = free | growth (Observe pricing tier).
Managed by /billing/webhook and /billing/checkout.

Layer 2 — Their Customers' Customers

End-users of the Observe customer's product

The people whose usage/costs/revenue the Observe customer tracks.

TableKey Columns
customerscustomer_id, name, email, stripe_customer_id
subscriptionssubscription_id, customer_id, plan_id, pricing_model
plansplan_id, name, price_amount
observe_eventscustomer_id, revenue_amount, cost_amount

Stripe context: The Observe customer's own Stripe account.
customers.stripe_customer_id = the end-user's Stripe ID (cus_*).
Connected via integrations table (their API key, not ours).
Managed by /integrations/stripe/connect and syncStripeDataForUser().

Where They Can Get Conflated

1. Column naming: users.stripe_customer_id (Layer 1) vs customers.stripe_customer_id (Layer 2) — same column name, completely different Stripe accounts.

2. Billing routes vs integration routes: /billing/* uses Tanso's Stripe keys. /integrations/stripe/* uses the user's encrypted key from integrations table. If code accidentally uses the wrong Stripe client, it hits the wrong account.

3. "customer" in code: In routes, "customer" sometimes means the Observe user (billing context) and sometimes means their end-user (analytics context). The customers table only holds Layer 2 data, but users/accounts also have customer-like fields.

Data Flow

LAYER 1: Observe's own billing
User signs up -> Clerk auth -> users + accounts + user_accounts
User upgrades -> /billing/checkout -> Tanso's Stripe -> webhook -> accounts.stripe_plan = 'growth'

LAYER 2: User's customer analytics
User connects Stripe -> /integrations/stripe/connect -> their API key encrypted -> integrations
Sync runs          -> syncStripeDataForUser() -> their Stripe API -> customers + subscriptions + plans
SDK event arrives  -> /events/ingest -> auto-create customers row -> enrich revenue from subscriptions
Name resolution   -> customers.stripe_customer_id -> their Stripe API -> update customers.name, email
Revenue backfill  -> runRevenueBackfill() -> re-enrich observe_events.revenue_amount

Revenue Enrichment Pipeline (the bridge)

SDK event arrives: customerReferenceId = "flatco", meta.stripe_customer_id = "cus_abc"

Step 1: Build bridge map (app ID → Stripe ID)
  "flatco" → "cus_abc"
  Sources: meta.stripe_customer_id, customers.stripe_customer_id, or cus_* detection

Step 2: Look up subscription by Stripe ID
  subscriptions WHERE customer_id = "cus_abc" → found: metered, unit_price = 0.01

Step 3: Calculate revenue
  metered:  unit_price × usage_units
  tiered:   tierPrice(mtdUsage) × usage_units
  flat:     $0 per event (MRR is fixed)

Step 4: Store on event
  observe_events.revenue_amount = calculated value
  observe_events.revenue_source = "per_unit" | "tiered" | "subscription"

Step 5: Resolve customer name (async)
  customers.stripe_customer_id = "cus_abc"
  → stripe.customers.retrieve("cus_abc")
  → UPDATE customers SET name = "FlatCo Inc", email = "..."
