Revenue Enrichment Pipeline

SDK event arrives: customerReferenceId = "flatco", meta.stripe_customer_id = "cus_abc"

Step 1: Build bridge map (app ID → Stripe ID)
  "flatco" → "cus_abc"
  Sources: meta.stripe_customer_id → customers.stripe_customer_id → cus_* detection

Step 2: Look up subscription by Stripe ID
  subscriptions WHERE customer_id = "cus_abc" → metered, unit_price = 0.01

Step 3: Calculate revenue
  metered:  unitPrice × usageUnits → revenue_source = "per_unit"
  tiered:   tierPrice(mtdUsage) × usageUnits → revenue_source = "tiered"
  flat:     $0 per event → revenue_source = "subscription"

Step 4: Store on event
  observe_events.revenue_amount, observe_events.revenue_source

Step 5: Resolve customer name (async)
  customers.stripe_customer_id → stripe.customers.retrieve() → update name + email

Customer Model: Two Layers

Layer 1 — Observe's Customers

Companies that sign up and pay for Observe.

TableKey Columns
usersemail, visitor_id, stripe_customer_id, stripe_plan
accountsname, clerk_org_id, stripe_customer_id, stripe_plan
user_accountsuser_id, account_id, role

Stripe: Tanso's own account. Routes: /billing/*

Layer 2 — Their Customers' Customers

End-users of the Observe customer's product.

TableKey Columns
customerscustomer_id, name, email, stripe_customer_id
subscriptionssubscription_id, customer_id, pricing_model
observe_eventscustomer_id, revenue_amount, cost_amount

Stripe: user's own account. Routes: /integrations/stripe/*

Conflation risks:
1. users.stripe_customer_id (Layer 1) vs customers.stripe_customer_id (Layer 2) — same name, different Stripe accounts
2. /billing/* uses Tanso keys; /integrations/stripe/* uses user's key — wrong client = wrong account
3. "customer" in code sometimes means Observe user (billing), sometimes end-user (analytics)
