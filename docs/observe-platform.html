Revenue Enrichment Pipeline

SDK event arrives: customerReferenceId = "flatco", meta.stripe_customer_id = "cus_abc"

Step 1: Build bridge map (app ID → Stripe ID)
  "flatco" → "cus_abc"
  Sources: meta.stripe_customer_id → customers.stripe_customer_id → cus_* detection

Step 2: Look up subscription by Stripe ID
  subscriptions WHERE customer_id = "cus_abc" → metered, unit_price = 0.01

Step 3: Calculate revenue
  metered:  unitPrice × usageUnits → revenue_source = "per_unit"
  tiered:   tierPrice(mtdUsage) × usageUnits → revenue_source = "tiered"
  flat:     $0 per event → revenue_source = "subscription"

Step 4: Store on event
  observe_events.revenue_amount, observe_events.revenue_source

Step 5: Resolve customer name (async)
  customers.stripe_customer_id → stripe.customers.retrieve() → update name + email

Customer Model: Two Layers

Layer 1 — Observe's Customers

Companies that sign up and pay for Observe.

TableKey Columns
usersemail, visitor_id, stripe_customer_id, stripe_plan
accountsname, clerk_org_id, stripe_customer_id, stripe_plan
user_accountsuser_id, account_id, role

Stripe: Tanso's own account. Routes: /billing/*

Layer 2 — Their Customers' Customers

End-users of the Observe customer's product.

TableKey Columns
customerscustomer_id, name, email, stripe_customer_id
subscriptionssubscription_id, customer_id, pricing_model
observe_eventscustomer_id, revenue_amount, cost_amount

Stripe: user's own account. Routes: /integrations/stripe/*

Conflation risks:
1. users.stripe_customer_id (Layer 1) vs customers.stripe_customer_id (Layer 2) — same name, different Stripe accounts
2. /billing/* uses Tanso keys; /integrations/stripe/* uses user's key — wrong client = wrong account
3. "customer" in code sometimes means Observe user (billing), sometimes end-user (analytics)

Bug Postmortems

Root cause analysis so these patterns don't repeat.

PM-1: Events page filters did nothing
FieldDetail
Root CauseVue 3 compiled @update:model-value="onSelectUpdate((v) => (selectedFeature = v))" as an inline handler: $event => onSelectUpdate(...). The HOF returned a function, but $event (the selected value) was never passed to it.
ImpactAll 4 dropdown filters on the Events page were non-functional. Inline table clicks still worked (they set refs directly).
FixPre-created named handler constants (onFeatureChange, etc.) so Vue uses them as method handlers, receiving the emitted value directly.
PreventionNever use HOF calls as inline Vue event handlers. Use named constants or arrow functions with explicit $event.
PM-2: Revenue enrichment broken after customer ID change
FieldDetail
Root CausecustomerReferenceId used to BE the Stripe customer ID (cus_*). It changed to an app-level ID with the Stripe ID moved to meta.stripe_customer_id. The subscription lookup, MTD usage query, and name resolution all joined on customer_id directly — assuming it was the Stripe ID.
ImpactRevenue = $0 on all events. Customer names not resolved. Stripe data synced but never linked to events.
FixAdded stripe_customer_id column to customers table. Built an appToStripe bridge map during ingest. Subscription lookup, MTD query, and name resolution now route through the bridge. Backfill populates stripe_customer_id from event meta for existing records.
PreventionWhen changing what a field means (customer_id from Stripe ID to app ID), audit every query that uses that field. The SYSTEM-MAP.md definitions table now clarifies the distinction.
PM-3: Silent catches hiding real errors
FieldDetail
Root CauseMultiple catch {} blocks with no logging: toggleEvent in EventsPage.vue, getStripeClientForUser in backfill. When these failed, the user saw no error — just empty or missing data.
ImpactEvent detail showed "Failed to load" with no clue why. Backfill silently skipped all customer name resolution if Stripe client failed for any reason (not just "no connection").
FixAll catches now console.error the actual error. Frontend shows error message with retry on re-expand.
PreventionRule: .claude/rules/no-silent-catches.md. Every catch must either log or propagate. No exceptions.
PM-4: Customer records never created on ingest
FieldDetail
Root CauseThe ingest route auto-created feature_definitions for new feature keys but had no equivalent for customers. The customers table was only populated by Stripe sync or CSV upload.
ImpactLEFT JOIN to customers returned NULL names. resolveStripeCustomerNames found no records to update. Events showed raw customer_id instead of resolved names.
FixAdded auto-create block after feature_definitions. Uses ON CONFLICT (user_id, customer_id) DO UPDATE SET stripe_customer_id = COALESCE(...) to fill in stripe_customer_id on subsequent events without overwriting existing names.
PreventionWhen adding auto-create for one entity type, check if related entities need the same treatment.
PM-5: Stripe sync failed — dual unique constraint on customers table
FieldDetail
Root Causecustomers has two unique constraints: UNIQUE(user_id, customer_id) and idx_customers_account_customer(account_id, customer_id). When Stripe sync inserts a customer that matches BOTH constraints, PostgreSQL can't resolve the ambiguity with ON CONFLICT (user_id, customer_id) — the second constraint also fires.
ImpactStripe sync fails entirely. No customers, subscriptions, or plans get updated. Revenue enrichment has stale data.
FixSwitched to delete-then-insert pattern for Stripe-synced customers (matching the memory rule: "Stripe sync deletes first, fresh insert carries names"). DELETE only removes rows where stripe_customer_id = customer_id (Stripe-created) or IS NULL, preserving SDK-created customer records.
PreventionNever use ON CONFLICT on tables with dual unique constraints. Use delete-then-insert or SELECT-first pattern. See bugs_upsert_dual_unique.md memory.
PM-6: Customer names not showing on Cohorts page
FieldDetail
Root CauseAPI returns customer_name but template checks c.name (undefined). Also, Clerk org org_3CgKJRdLwG8qTdZFdIGbviyJn7e had no matching account in DB — created pre-Clerk migration without clerk_org_id.
ImpactAll customers showed raw cus_* IDs instead of resolved names. Stripe sync failed because accountId resolved to null.
FixFixed c.name → c.customer_name in 3 places. Linked Clerk org to existing account 82 in prod DB. Added email display under name.
PreventionWhen field names differ between API and frontend types, catch it in code review. Run Clerk org → account migration for all pre-Clerk accounts.
PM-7: Revenue rule — flat subs show $0 per event (by design)
FieldDetail
RuleRevenue at the customer level (MRR from subscription) is the right signal. Revenue at the event/feature level is misleading for flat subs — proportional attribution gives every feature the same margin %.
Where to show MRRCustomer detail page, Cohorts page MRR column, Analytics by-customer tab. NOT per-event or per-feature.
Per-event revenueOnly meaningful for metered (unitPrice × usage) and tiered pricing. Flat subs = $0 per event, revenue_source = "subscription".
Fix appliedCohorts page MRR column now shows actual dollar amount from subscriptions instead of movement badge. Removed "Unprofitable" cohort label badge. Margin calc updated (4a945c9): cohorts endpoint now falls back to subRevenueMap (subscription MRR) when event-level revenue is $0, so flat-sub customers show correct ~100% margin instead of -100%.
PM-8: Pre-Clerk accounts not linked — users get empty accounts on login
FieldDetail
Root CauseUsers created before Clerk migration have UUID visitor_id values. When they log in via Clerk, ensureVisitor creates a new user record (or updates via ON CONFLICT email). But the old account has no clerk_org_id, so the Clerk org can't find it. A new empty account gets created instead.
AffectedOnly pre-Clerk users (4 users with UUID visitor_ids). New signups via Clerk work correctly. GrowthOS (post-Clerk) is fine.
ImpactUser sees empty Data Sources page with no API key, no events, no Stripe connection — even though their old account has all this data.
FixAdded auto-backfill in ensureVisitor: when a Clerk user's account has no clerk_org_id, stamp the Clerk org ID on it. The ON CONFLICT (email) DO UPDATE SET visitor_id clause already updates the visitor_id — the auth flow just needs to complete (user can't bounce at the org setup screen).
PreventionRun a one-time migration to update all UUID visitor_ids to Clerk user IDs. Add monitoring for accounts with null clerk_org_id.
PM-9: Agent hallucinated variable name in cohorts query (DDS-3)
FieldDetail
Root CauseDDS-3 agent added pricingModelMap[customerId] to the cohorts response builder, but the loop variable is row.customer_id, not customerId. The agent hallucinated a variable name that doesn't exist in scope.
SymptomGET /cohorts returns 500 "Failed to compute cohorts" with ReferenceError: customerId is not defined. Cohorts page fails to load for logged-in users.
ImpactCohorts page completely broken for authenticated users. Guest mode unaffected (uses client-side sample data).
FixpricingModelMap[customerId] → pricingModelMap[row.customer_id] at cohorts.ts:532. Also added detail: String(error) to the 500 response so future errors are diagnosable from the browser.
PreventionAgent swarm cards should specify exact variable names in scope, not just file + line number. Post-merge testing with a test client catches ReferenceErrors that type-checking misses (TypeScript doesn't catch runtime ReferenceError in template expressions or dynamic map lookups).
